# Blueprint (`/marketing-stack blueprint`)

Regenerate `marketing/infrastructure.md` from the observed state of installed recipes + the strategy.

## Phase 1 — Gather

Same gather step as inventory mode. Read:

- All recipe detection signals (see `steps/inventory.md` Phase 2)
- Existing `marketing/infrastructure.md` if present (to preserve Notes and last-verified dates)
- `marketing/07_channels.md` for strategy-alignment column
- `marketing/activities.md` for the activity references

## Phase 2 — Build the table

For each of the 17 recipes, compute:

- Status: ready / setup-pending / not-installed-needed / not-installed-nice-to-have / blocked / out-of-scope
- Last verified: carry forward from prior blueprint if status is still `ready`, otherwise today
- Notes: carry forward any free-text notes from prior blueprint

## Phase 3 — Optionally run validation

For recipes marked `ready`, offer to run validation calls in batch:

> "Blueprint regenerates from observed state. Want to re-verify ready recipes by running their validation calls? (takes ~1 min; (y/n))"

If yes, run each recipe's validation call from `recipes/<name>/RECIPE.md`, update status and last-verified accordingly.

If no, trust prior state.

## Anti-patterns

- **Re-writing without preserving notes.** Blueprint regenerates structure but carries forward human-edited notes. Never blow them away.
- **Skipping the strategy-alignment table.** The whole point of the blueprint is to make gaps vs strategy visible. Don't omit it.
- **Generating when nothing changed.** If the observed state exactly matches the prior blueprint, tell the user "no changes detected; blueprint untouched."
- **Hiding blocked recipes.** If something's blocked, surface it explicitly in Open issues + Next actions.
